What it simulates

The mock server answers real HTTP requests exactly as Cubi would. Your app doesn't need to know it's talking to a local fake.

  • POST/security/v1/oauth2/token
  • GET/accounts/v1/
  • GET/accounts/v1/{account_id}/transactions
  • POST/ach/v1/outgoing/debit
  • POST/ach/v1/outgoing/credit
  • POST/wires/v1/outgoing
  • GET/ach/v1/outgoing/{payment_id}
  • GET/wires/v1/outgoing/{payment_id}
  • GET/health

How to connect your app

Start the mock, then copy the Env Bundle from the browser UI into your app's .env.local file. The two values that matter most are:

CUBI_BASE_URL=http://127.0.0.1:8791
CUBI_TOKEN_URL=http://127.0.0.1:8791/security/v1/oauth2/token

State persistence

Payments you create survive server restarts. The state is saved in .runtime/mock_cubi_state.json. Use Reset State in the UI whenever you want a clean slate.

What is MCP?

MCP (Model Context Protocol) is an open standard that lets AI assistants call external tools. Once you connect this playground to your AI tool, it gains 10 actions it can call — letting it start the mock, browse accounts, create payments, and more, just from a plain-language request.

Step 1 — Get your config snippet

Start the playground (see setup steps below), start the mock from the UI, then click Copy in the MCP Config panel. The snippet has your exact file paths already filled in and looks like this:

{
  "mcpServers": {
    "cubi-mock-playground": {
      "command": "C:\\...\\cubi-mcp-playground\\.venv\\Scripts\\python.exe",
      "args": ["-m", "server.mcp_server"],
      "cwd": "C:\\...\\cubi-mcp-playground"
    }
  }
}

Use the snippet from the UI — it has your real paths. The above is just an example of the shape.

Step 2 — Add to your AI tool

In your terminal, run the claude mcp add command. Replace the paths with the values from the MCP Config panel in the UI:

claude mcp add cubi-mock-playground -s user -- ".venv\Scripts\python.exe" -m server.mcp_server

Run this command from inside the cubi-mcp-playground folder. The -s user flag makes it available across all your projects.

claude mcp add cubi-mock-playground -s user -- ".venv/bin/python" -m server.mcp_server

Run this command from inside the cubi-mcp-playground folder. The -s user flag makes it available across all your projects.

Alternatively, paste the JSON snippet from the MCP Config panel directly into .claude.json at the repo root, or into ~/.claude.json for global access.

💡

After adding, type /mcp in Claude Code to confirm the server shows as connected.

Open the Claude Desktop config file in a text editor:

%APPDATA%\Claude\claude_desktop_config.json

Paste the snippet from the MCP Config panel into the "mcpServers" block. If the file is empty, paste the full JSON including the outer braces. Save the file and restart Claude Desktop.

~/Library/Application Support/Claude/claude_desktop_config.json

Paste the snippet from the MCP Config panel into the "mcpServers" block. If the file is empty, paste the full JSON including the outer braces. Save the file and restart Claude Desktop.

💡

After restarting, look for the hammer icon (🔨) in the chat input bar — that confirms MCP tools are available.

Open Cursor and navigate to:

Settings → Cursor Settings → Features → MCP Servers → Add new MCP server

Paste the snippet from the MCP Config panel into the config field. Cursor will automatically detect and connect to the server. No restart required.

💡

The mock must be running (started from the browser UI) before Cursor can call its tools. Start the mock first, then ask Cursor to use it.

For AI tools or MCP clients that prefer an HTTP endpoint instead of a local process, run the MCP server in HTTP mode:

powershell -ExecutionPolicy Bypass -File .\scripts\start-mcp-http.ps1
.venv/bin/python -m server.mcp_server --transport streamable-http --host 127.0.0.1 --port 8811

The MCP endpoint will be available at:

http://127.0.0.1:8811/mcp

Point your MCP client at that URL using the streamable-HTTP transport setting.

Before you begin

You need one thing installed: Python 3.12 or newer

Open a terminal and run the command below to check. If the number shown is 3.12 or higher, you're good.

python --version
💡

If the version is below 3.12 or you see "python is not recognized", download the latest Python from python.org. On Windows, check "Add Python to PATH" during installation.

Everything else (PowerShell on Windows, Terminal on Mac) comes pre-installed with your operating system.

Navigate into the repo folder

Create a virtual environment

This creates an isolated Python workspace so nothing affects the rest of your machine. Run once, never again.

python -m venv .venv

You'll see a progress line ending with "Successfully installed mcp…". Setup is now complete.

Open the UI in your browser

Navigate to:

Start the mock from the UI

In the Mock Runtime panel:

  • Leave Bind Host as 127.0.0.1

The status badge flips from offline to online and the other panels fill in automatically.

Choosing a profile

Profiles control what data and payment outcomes the mock generates.

ProfileBest for
defaultGeneral use — balanced mix of accounts and normal payment progression
returnsDemoing ACH return scenarios (payments that come back after being sent)
repairTesting the "pending repair" state — a payment that gets flagged and needs fixing
wire-heavyEnvironments that mostly send wires rather than ACH payments
⚠️

When switching profiles, check "Reset state on start" in the UI before clicking Start Mock. That clears payments from the previous session so the new profile loads cleanly.

Connecting your app to the mock

Copy the Env Bundle from the UI and save it as a local override file for your app (e.g. .env.local):

CUBI_MODE=real
CUBI_BASE_URL=http://127.0.0.1:8791
CUBI_TOKEN_URL=http://127.0.0.1:8791/security/v1/oauth2/token
CUBI_CLIENT_ID=mock-client
CUBI_CLIENT_SECRET=mock-secret
CUBI_ACCOUNTS_PATH=/accounts/v1/
CUBI_TRANSACTIONS_PATH_TEMPLATE=/accounts/v1/{account_id}/transactions
💡

Notice CUBI_MODE=real. Your app stays in its normal HTTP mode — it just points at localhost. No special mock-only code paths are needed.

Common problems

? The UI loads but the mock badge says "offline"
Normal on first load. The playground web server is up but the mock hasn't been started yet. Click Start Mock in the Mock Runtime panel.
? Start Mock fails immediately
Port 8791 may already be in use. Change it to 8795 in the UI and try again. Copy the updated Env Bundle so your app gets the right URL.
? The playground script won't run — "running scripts is disabled"
The command from Step 5 already includes -ExecutionPolicy Bypass. Make sure you're pasting the full command, not just the script filename on its own.
? My app is still hitting the real Cubi sandbox
The app didn't pick up the local env file. Confirm CUBI_BASE_URL is http://127.0.0.1:8791 in the running process. Most frameworks need a full restart to reload env files.
? I want a clean slate for a demo
Check "Reset state on start" in the Mock Runtime panel, then click Start Mock again. All previous payments are wiped and fresh seed data loads.
? "python" is not recognized / command not found
On Windows try py; on Mac try python3. If neither works, reinstall Python from python.org — on Windows check "Add Python to PATH" during installation.
? My AI assistant says it can't find the MCP tools
The MCP server connects at startup. Make sure the playground is running and the mock is started, then restart your AI tool so it re-connects to the MCP server. In Claude Code, run /mcp to verify the connection status.
